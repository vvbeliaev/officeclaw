# Sync/Async Vertical Framework

*Март 2026 | На основе brainstorm-сессии*

---

## TL;DR

Одна вертикаль = один живой репозиторий. Operator + Claude Code образуют sync контур, nanobot агент — async контур. Repo — коммуникационный медиум между ними. CC читает traces агента и через диалог с оператором инжектирует WM обратно в repo. Это не новая технология — это convention over configuration для агентных систем.

---

## 1. Два контура, одно repo

```
[Sync контур]                      [Async контур]
Operator + Claude Code  ←——— REPO ———→  nanobot agent
     IDE / SSH                               (server)
```

**Async агент знает только:**
- Repo structure (state, artifacts, skills)
- Вертикальную логику
- Skills (text injection в контекст)

**Async агент не знает:**
- Про оператора, CC, блокинги
- Про механику нотификаций
- Про инфраструктуру sync контура

**Repo — не просто хранилище.** Это кристаллизация WM оператора которую агент использует в его отсутствие. Каждый файл который оператор трогает с CC — инъекция WM. Нет отдельного "world_model.md" — repo IS world model.

## 2. Living Repo — анатомия вертикали

```
myvertical/
├── .nanobot/
│   ├── config.yaml      ← cron, channels, model (оператор пишет)
│   └── AGENTS.md        ← system prompt агента (оператор пишет)
│
├── skills/              ← WM оператора в действии (оператор пишет)
│   ├── skill_name.md    ← text injection, вербализует как делать
│   └── ...
│
├── contexts/            ← активная работа (агент пишет / оператор правит)
│   └── {id}/
│       ├── state.md     ← snapshot начала и конца каждого прогона
│       └── artifacts/
│
├── traces/              ← история прогонов (агент пишет, CC читает)
│
└── README.md            ← ориентация при открытии (оператор пишет)
```

**Git = бизнес-лог.** `git log` = история работы вертикали. `git diff` = дельта между прогонами.

## 3. V1 Loop — как на самом деле работает

Async агент в первых версиях минимален. Он не думает про оператора — просто хорошо записывает что делает.

```
1. Async агент запускается
   → делает работу вертикали
   → traces собираются автоматически runtime
   → state snapshot: before/after прогона

2. Оператор открывает IDE + CC
   → CC анализирует traces: что произошло, где проблемы
   → "Вот что агент делал. Вот где спорные решения. Что менять?"

3. Диалог → понимание проблемы
   → оператор объясняет что не так, что хотел видеть
   → вместе формулируют что изменить
   → это и есть WM injection — в форме разговора, не документа

4. CC редактирует repo
   → AGENTS.md, skills/*.md, contexts/*/state.md
   → изменения landing в repo

5. → следующий прогон агента с улучшенными инструкциями
```

**Ключевой сдвиг:** CC — не инструмент написания кода, а инструмент чтения async процесса и WM injection через диалог.

## 4. Что реально нужно спроектировать (backbone конвенции)

Это Tech Framework — convention over configuration. Аналог Rails: не новая технология, а конвенция которая делает цикл самоподдерживающимся.

### Что нужно в v1

**Trace format** — traces должны быть читаемы для CC как бизнес-нарратив, не только технические логи. Каждый значимый шаг агента должен быть понятен оператору без знания деталей реализации.

**State snapshot** — зафиксированный state в начале и конце каждого прогона. Это дельта = что изменилось за один запуск. Основа для анализа регрессий.

**Skills convention** — формат skill файла: когда активируется, какие шаги, как CC его улучшает на основе trace анализа. Skills = способ кристаллизовать WM оператора в повторяемое поведение агента.

**Repo structure** — стандартная папочная структура чтобы CC мог навигировать любую вертикаль без объяснений.

### Что не нужно в v1

- Отдельный файл "world_model.md"
- Специальный attention/ протокол
- Reflection skill в async агенте
- ofclaw CLI
- Blocking механизм

Всё это может вырасти из практики когда станет понятно что реально нужно.

## 5. Честная оценка рисков

**World model bloat** — оператор добавляет, редко чистит. Skills и AGENTS.md могут накопить противоречия. Нужна дисциплина периодической ревизии с CC.

**Context selection** — нельзя загружать весь repo в контекст агента. Нужна селекция: что из skills и state релевантно конкретному прогону. Это нетривиально без правильных конвенций.

**Operator fatigue** — если CC анализ после каждого прогона не приносит очевидной пользы, оператор перестаёт делать sync сессии. Система без инъекций деградирует в slop.

**Reflection quality** — если добавлять reflection step в агента: LLM не всегда честен с собой. Ложная уверенность в reflection хуже чем её отсутствие.

---

## 6. Что это не решает само по себе

**Нет механизма "оживления"** — когда оператор давно не заходил, агент не знает что делать по-другому. Нужно либо operator присутствие, либо явный decay механизм.

**Нет стандартного bootstrap** — новая вертикаль с пустым repo: откуда берётся начальный WM? Нужен bootstrap protocol.

**Нет формального способа сказать "это заблокировано"** — в v1 агент просто записывает неуверенность в traces. Оператор увидит при следующей CC-сессии. Это latency которую нужно принять.

## 8. Минимальный эксперимент

Взять digital-agency вертикаль. Добавить:
1. Структурированный trace format в AGENTS.md (что агент должен писать в traces/)
2. State snapshot: агент пишет state.md до и после каждого прогона
3. Skills с явной конвенцией (заголовок, условие активации, шаги)

После 5–10 прогонов: открыть с CC, попросить "проанализируй traces, что менять в инструкциях?".

Метрика: стал ли CC-анализ приносить конкретные изменения в skills/AGENTS.md? Если да — flywheel работает.
